Learning ML #4: Tiny Character-Level RNN on Shakespeare

April 6, 2026
14 min read
By Declan Porter

This was the most complex project so far — training a character-level language model from scratch. Given a sequence of characters, the model learns to predict the next character. After training on a corpus of Shakespeare, it can continue a prompt in a vaguely Shakespearean style. It's bad at first. Watching it get less bad is satisfying.

What Is a Character-Level Language Model?

The task: given the characters h e l l, predict o. Given e l l o, predict the next character. Repeat across an entire corpus.

Why characters instead of words? It's simpler to implement (the vocabulary is just the unique characters in the text — usually around 65–100 items) and it teaches the core ideas without the complexity of tokenisation algorithms like BPE.

From Text to Tokens

The pipeline starts by reading the full text corpus and building a character vocabulary: 

unique_chars = sorted(set(content))   # deterministic ordering

vocab = {}
vocab_inv = {}
for i, c in enumerate(unique_chars):
    vocab[c] = i
    vocab_inv[i] = c

vocab_len = len(vocab)   # ~65 for Shakespeare

Using sorted(set(...)) is important for determinism — if the vocab assignment changes between runs, saved checkpoints become incompatible. Then the full text gets mapped to a list of integer IDs: 

tokens_list = [vocab[c] for c in content]
tokens = torch.tensor(tokens_list, dtype=torch.long)

80% becomes training tokens, 20% validation. The dtype must be torch.long (int64) — embeddings and CrossEntropyLoss both require integer indices, not floats.

The Sliding Window Dataset

Rather than feeding one character at a time, the model receives a context window of T characters and predicts the next T characters (the same window shifted by one position): 

class TokenDataset(Dataset):
    def __init__(self, data_tensor, window_size):
        self.data = data_tensor
        self.window_size = window_size

    def __len__(self):
        return len(self.data) - self.window_size

    def __getitem__(self, idx):
        inputs = self.data[idx : idx + self.window_size]
        targets = self.data[idx + 1 : idx + self.window_size + 1]
        return (inputs, targets)

For example with window_size=4 and text "hello":

  • Input: [h, e, l, l] → Target: [e, l, l, o]

Every position in the window is simultaneously a prediction task. This is what gets flattened for the loss computation later.

The Model: Embedding + GRU + Linear

Before anything else: what is an embedding?

An nn.Embedding(vocab_size, embed_dim) is a learnable lookup table. Each token ID maps to a row vector of length embed_dim. Instead of feeding the raw integer 42 into a linear layer (which has no geometric meaning), you get a vector of 256 learned floats that the model can find meaningful. It's just a matrix lookup: token → row.

After embedding, the token sequence goes into a GRU (Gated Recurrent Unit). Unlike a plain linear layer, which processes each input in isolation, an RNN carries a hidden state forward through time:

ht = f(Wx·xt + Wh·ht−1 + b)

The GRU improves on vanilla RNN by adding update and reset gates that control how much past context to keep versus overwrite. This substantially helps with longer sequences where plain RNNs lose signal to the vanishing gradient problem. 

class TinyCharRNN(nn.Module):
    def __init__(self):
        super(TinyCharRNN, self).__init__()
        self.embed = nn.Embedding(vocab_len, 256)
        self.rnn1 = nn.GRU(256, 256, batch_first=True)
        self.rnn2 = nn.GRU(256, 256)
        self.linear = nn.Linear(256, vocab_len)

    def forward(self, x):
        l1 = self.embed(x)
        rnn_out1, h = self.rnn1(l1)
        rnn_out2, h = self.rnn2(rnn_out1)
        l3 = self.linear(rnn_out2)
        return l3

Shape flow through the model for a batch of size B and window T:

  • Token IDs: [B, T]
  • After embedding: [B, T, 256]
  • After GRU: [B, T, 256]
  • After linear: [B, T, vocab_len] — one logit vector per position

The batch_first=True flag is important — by default PyTorch's RNN layers expect shape [T, B, features], not [B, T, features]. Since the embedding output is batch-first, rnn1 needs the flag. Note that rnn2 receives the output of rnn1 which is also already in the right order.

Loss: Flattening for CrossEntropy

CrossEntropyLoss expects logits of shape [N, C] and targets of shape [N]. My output is [B, T, vocab_len] and targets are [B, T]. Flattening collapses the batch and time dimensions together — every position is treated as an independent classification task: 

training_preds = model(training_inputs)         # [B, T, V]
training_preds = torch.flatten(training_preds, 0, 1)  # [B*T, V]
training_targets = torch.flatten(training_targets, 0)  # [B*T]
loss = criterion(training_preds, training_targets)

Perplexity

For language models, cross-entropy loss in nats has a useful interpretation via perplexity:

perplexity = eloss

It's the model's "effective number of equally likely choices" per next character. A random model on a 65-character vocab would have loss ≈ ln(65) ≈ 4.17 and perplexity ≈ 65. After a few epochs of training: 

avg_t_loss=1.6852 | avg_v_loss=1.9097 | perplexity=6.75

Down from ~65 to ~6.75 — the model is genuinely learning structure in the text. It's uncertain among about 6–7 plausible next characters on average rather than 65.

Watching It Generate

After each epoch I ran a generation test using the model's predictions as new input — greedy decoding with argmax. The output after a few epochs on a CPU: 

---Test---
General:
Are we blind !?
Deploy the garrison!
Move!!!
     adr g
Ane ti seong tf
oaler the srleoehr:
are
tEOEII1

It picks up on punctuation patterns and capital letter usage surprisingly quickly. The actual words are mostly gibberish at this scale, but you can see the model has learned something about the structure of the text — newlines, colons after character names, short words.

This quality is expected for a small vanilla RNN trained for only a few epochs on CPU. The next steps to improve it: GRU (already done) → LSTM, longer context window, more hidden units, more epochs, and better generation (temperature sampling instead of greedy argmax).

What I Learned

  • Character-level models are a great way to learn sequence modelling without needing to understand subword tokenisation first.
  • Use sorted(set(...)) for deterministic vocab assignment — otherwise saved checkpoints are incompatible across runs.
  • nn.Embedding is just a learnable lookup table. Token ID → learned vector of length E.
  • RNN returns two outputs: the sequence of hidden states (one per time step) and the final hidden state. Use the former for per-token predictions.
  • Always pass batch_first=True if your tensors are batch-first, or transpose before the RNN.
  • Flatten [B, T, V] → [B*T, V] for CrossEntropyLoss. Every position is an independent classification.
  • Perplexity = e^loss. For reference: random on a 65-char vocab ≈ 65. A trained model on Shakespeare gets to ~6–7.
  • Greedy argmax generation is simple but repetitive. Temperature sampling produces more natural-looking text.
