Learning ML #3: MNIST — From MLP to CNN

April 6, 2026
12 min read
By Declan Porter

MNIST is the classic benchmark: 70,000 handwritten digit images, each 28×28 pixels, labelled 0–9. This was my first encounter with real image data, torchvision's dataset utilities, CrossEntropyLoss, and convolutional networks. I built an MLP first, then upgraded to a CNN to understand the difference.

Loading the Data with Torchvision

Torchvision downloads and caches MNIST automatically. You attach a transform — a preprocessing pipeline that runs on each image when the dataloader fetches it: 

transform = torchvision.transforms.Compose([
    torchvision.transforms.ToTensor(),
    torchvision.transforms.Normalize((0.5,), (0.5,)),
    torchvision.transforms.Lambda(lambda x: torch.flatten(x))
])

ToTensor() converts a PIL image to a float tensor and scales pixel values from [0, 255] to [0, 1]. Normalize((0.5,), (0.5,)) then shifts and scales values to roughly [−1, 1] — centred near zero for better gradient conditioning. The flatten step (for the MLP) turns the 28×28 image into a vector of 784 values.

Why not keep it in [0, 1]? Nothing is technically wrong with that — normalising to zero-mean is just an optimisation aid, especially for activations like Tanh that are most expressive near zero.

Part 1: MLP Baseline

The MLP takes the flattened 784-pixel vector and classifies it into one of 10 digit classes. It's deliberately overkill (784 hidden units) but validates that the training pipeline works before adding convolution complexity. 

class MLP(nn.Module):
    def __init__(self):
        super(MLP, self).__init__()
        self.input = nn.Linear(784, 784)
        self.input_act = nn.Tanh()
        self.hidden = nn.Linear(784, 784)
        self.hidden_act = nn.Tanh()
        self.output = nn.Linear(784, 10)

    def forward(self, x):
        l1 = self.input_act(self.input(x))
        l2 = self.hidden_act(self.hidden(l1))
        l3 = self.output(l2)
        return l3

The output layer has 10 neurons — one per digit. These outputs are raw logits, not probabilities. The loss function converts them.

CrossEntropyLoss — Not One-Hot

For 10-class classification I switched from BCEWithLogitsLoss to CrossEntropyLoss. One common mistake: thinking you need one-hot label vectors because your output has 10 values. You don't. CrossEntropyLoss takes:

  • Logits: shape [N, 10]
  • Labels: shape [N] of integer class IDs (dtype long)

Internally it applies softmax and computes negative log-likelihood. The math for why integer targets are sufficient is elegant. For one sample with logits z and true class y:

L = −zy + log(Σj ezj)

The gradient with respect to any logit zi is:

∂L/∂zi = pi − 𝟙[i = y]

Where pi is the softmax probability for class i. For the true class y this gradient is negative (push it up). For all other classes it's positive (push them down). So telling the model which class is correct is all it needs — the softmax competition handles the rest.

Accuracy for Multi-Class

Binary accuracy used sigmoid + round. Multi-class accuracy uses argmax — the index of the largest logit is the predicted class: 

def calc_accuracy(predictions, answers):
    pred_class = predictions.argmax(dim=1)
    equal = torch.eq(pred_class, answers).float()
    return torch.mean(equal)

Early result after one epoch: 

t_loss=0.2249 | v_loss=0.1706 | v_acc=0.9219

~92% validation accuracy from an MLP after one epoch on MNIST is very plausible. It confirms the pipeline is correct and gives a baseline to beat with the CNN.

Part 2: Convolutional Network

An MLP treats an image as a flat vector — it has no concept that nearby pixels are spatially related. A CNN exploits that structure by applying learned filters that slide across the image looking for local patterns (edges, corners, strokes).

My CNN uses two convolutional blocks, each with Conv2d, ReLU, and MaxPool2d, followed by a single linear output layer: 

class Conv(nn.Module):
    def __init__(self):
        super(Conv, self).__init__()
        self.conv1 = nn.Conv2d(1, 32, 3)   # [1,28,28] -> [32,26,26]
        self.conv1_act = nn.ReLU()
        self.conv1_pool = nn.MaxPool2d(2)   # -> [32,13,13]
        self.conv2 = nn.Conv2d(32, 64, 3)   # -> [64,11,11]
        self.conv2_act = nn.ReLU()
        self.conv2_pool = nn.MaxPool2d(2)   # -> [64,5,5]
        self.output = nn.Linear(1600, 10)   # 64 * 5 * 5 = 1600

    def forward(self, x):
        c1 = self.conv1_pool(self.conv1_act(self.conv1(x)))
        c2 = self.conv2_pool(self.conv2_act(self.conv2(c1)))
        f = torch.flatten(c2, 1)
        return self.output(f)

Tracking the spatial dimensions through the network is critical to getting the flattened size right for the output layer. With a 3×3 kernel and no padding:

  • Conv with kernel 3: H − (3 − 1) = H − 2
  • MaxPool with kernel 2: H // 2

So: 28 → 26 → 13 → 11 → 5. Final feature map: 64 channels × 5 × 5 = 1600 values per image.

The CNN uses ReLU rather than Tanh for activations. ReLU is standard in convolutional layers — it's computationally cheap and doesn't saturate like Tanh does for large values.

What I Learned

  • Transforms in torchvision are a clean way to chain preprocessing — they run lazily on each fetch, not upfront on the whole dataset.
  • CrossEntropyLoss expects integer label indices, not one-hot vectors. This is by design — the softmax competition is implicit in the math.
  • Multi-class accuracy = argmax, not sigmoid. Using sigmoid on a 10-class output is wrong.
  • Spatial dimension tracking for CNNs: output size = input − (kernel − 1) with no padding, then halved by MaxPool2d(2).
  • Build an MLP baseline first. It validates your pipeline before you add architectural complexity.
