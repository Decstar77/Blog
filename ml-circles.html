Learning ML #2: Binary Classification on Circles

April 6, 2026
10 min read
By Declan Porter

After regression, the next logical step is classification. I used scikit-learn's make_circles to generate a synthetic 2D dataset — two concentric rings, each a different class — and trained an MLP to separate them. This taught me logits, BCE loss, accuracy metrics, and what overfitting actually looks like.

The Dataset

sklearn.datasets.make_circles generates points in 2D space forming two concentric rings. Every point has coordinates (x₁, x₂) and a label — 0 for the outer ring, 1 for the inner ring. It's a purely synthetic dataset, generated on the fly, no downloading required. 

noisy_circles_x, noisy_circles_y = make_circles(
    n_samples=2000, noise=0.1, random_state=6
)

2000 samples with a noise factor of 0.1 — enough to make it non-trivial without being impossible. Input shape: [N, 2]. Labels: [N] of 0s and 1s.

Loss Function: Why Not MSE?

For regression I used MSE. For binary classification the right loss is BCEWithLogitsLoss. Here's why the distinction matters.

The model outputs a raw float — a logit — not a probability. A logit can be any real number. BCEWithLogitsLoss internally applies sigmoid to convert the logit to a probability, then computes binary cross-entropy:

BCE(z, y) = −[y · log(σ(z)) + (1−y) · log(1−σ(z))]

Doing the sigmoid inside the loss (rather than in the model) is numerically more stable. And crucially — during training you never round or threshold the output. Rounding kills gradients; the model needs soft probabilities to know which direction to move.

The Model

Two inputs (x₁, x₂), two hidden layers of width 64, one output neuron (the logit): 

class Classifier(nn.Module):
    def __init__(self):
        super(Classifier, self).__init__()
        size = 64
        self.input = nn.Linear(2, size)
        self.input_act = nn.Tanh()
        self.hidden = nn.Linear(size, size)
        self.hidden_act = nn.Tanh()
        self.output = nn.Linear(size, 1)

    def forward(self, x):
        res_1 = self.input_act(self.input(x))
        res_2 = self.hidden_act(self.hidden(res_1))
        res_3 = self.output(res_2)
        return res_3

Overfitting: Seeing It Happen Live

Early in training, with only a handful of samples, I saw this: 

Training loss=0.4538 | Validation loss=0.7188

Training loss was going down, validation loss was going up. That's textbook overfitting — the model was memorising the training points rather than learning the general ring structure.

The fixes, in order of effectiveness:

  • More data — I bumped up to 2000 samples. This was the biggest win.
  • Reduce model capacity if the model is too large for the dataset.
  • Weight decay via AdamW — the decoupled shrinkage discourages large weights.
  • Early stopping based on validation loss.

After these fixes: 

Training loss=0.3468 | Validation loss=0.3598 | Validation acc = 0.8250

Train and validation losses tracking together — healthy. ~82.5% accuracy on noisy circles is solid.

Accuracy: What It Is and How to Compute It

Loss is a continuous signal the model optimises. Accuracy is the human-readable metric: the fraction of predictions that are correct after thresholding.

For BCEWithLogitsLoss there's a handy shortcut. Instead of applying sigmoid and rounding, you can just check the sign of the logit — a positive logit maps to class 1, negative to class 0: 

def calc_accuracy(predictions, answers):
    probabilities = torch.sigmoid(predictions)
    guesses = torch.round(probabilities)
    total = torch.numel(guesses)
    num_correct = total - torch.count_nonzero(guesses - answers)
    return num_correct / total

Two models can have the same accuracy but different losses. The one with lower loss is outputting more confident correct predictions — further from the 0.5 threshold. Loss captures quality; accuracy just captures correctness.

BCE Baseline: The Random Guess Number

For roughly balanced classes, a random model outputs logits near 0, giving a BCE loss of approximately ln(2) ≈ 0.693. If your loss is above that after training, your model is worse than a coin flip. If it's well below, it's learning.

My model started around 0.69 and converged to ~0.35 — roughly half the random-guess loss, which is a meaningful improvement on a noisy problem.

What I Learned

  • BCEWithLogitsLoss = sigmoid + BCE in one numerically stable operation. Don't apply sigmoid yourself before the loss.
  • Overfitting has a clear visual signature: training loss down, validation loss up.
  • More data is usually the cheapest fix for overfitting.
  • Accuracy and loss measure different things. Track both.
  • For BCE loss: random guessing ≈ 0.693. Use this as your sanity-check baseline.
